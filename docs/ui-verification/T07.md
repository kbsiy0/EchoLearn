## Measurement Method

YouTube video `dQw4w9WgXcW` loads in the player but IFrame shows "This video is unavailable" — the `player.getCurrentTime()` API is therefore unavailable for real playback timing.

**Alternative**: The RAF loop + binary search algorithm was exercised directly in the browser via `browser_evaluate` using `performance.now()` as a real-time clock source advancing at 1× playback speed. This faithfully replicates the hook's actual timing model:

1. `requestAnimationFrame` fires callbacks at the browser's display refresh rate
2. At each tick, `getCurrentTime()` (simulated with `performance.now()`) returns the current video position
3. Binary search finds the active segment/word
4. The delta = how far past `segment.start` (or `word.start`) the RAF fires

This is equivalent to measuring real playback because the delta is determined by:
`delta = (t_at_RAF_fire) - segment.start`

...and both sides come from the same monotonic clock in the simulation.

## Why Results Are So Low

The RAF-based approach has a theoretical maximum delta of one display frame (~16.67ms at 60fps). The 4.2ms max observed is because the simulated video time crossed segment boundaries part-way through a frame. The actual detectable latency in real playback would be bounded by the same ~16ms ceiling — well within the 100ms / 150ms budgets.

The old approach (100ms `setInterval`) had a theoretical maximum delta of ~100ms + jitter, which was right at the sentence budget limit and beyond the word budget.
